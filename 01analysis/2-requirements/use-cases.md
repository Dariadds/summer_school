## UC-001: Просмотр расписания заездов

| Поле | Значение |
| :-- | :-- |
| **ID** | UC-001 |
| **Приоритет** | P1 |
| **Источник** | [DD], [CQ:6], [CQ:33] |
| **Связанные ФТ** | FT-001, FT-002, FT-003 |
| **Актёр** | Клиент |
| **Предусловия** | Приложение установлено; данные расписания доступны через API |
| **Триггер** | Клиент хочет найти подходящий заезд |

### Основной поток

1. Клиент открывает экран расписания.
2. Система отображает список заездов на ближайшие 7 дней.
3. (Опционально) Клиент задаёт фильтр по диапазону дат.
4. Система обновляет список заездов.

### Альтернативные потоки

- **A1. Нет заездов.** Система показывает сообщение «Пока нет доступных заездов».

### Постусловия

- Клиент видит актуальный список заездов; данные расписания не изменяются.

## UC-004: Бронирование мест на заезд

| Поле | Значение |
| :-- | :-- |
| **ID** | UC-004 |
| **Приоритет** | P1 |
| **Источник** | [DD], [CQ:3], [CQ:10–15], [CQ:33] |
| **Связанные ФТ** | FT-009–FT-016 |
| **Актёр** | Клиент |
| **Предусловия** | Клиент авторизован; выбранный заезд имеет свободные места |
| **Триггер** | Клиент хочет записаться на заезд |

### Основной поток

1. Клиент выбирает заезд.
2. Клиент указывает количество мест (1–4).
3. Для каждого места клиент выбирает аренду экипировки (шлем, подшлемник) или использование своей.
4. Система проверяет доступность мест и прокатной экипировки через API.
5. Система показывает экран подтверждения с итоговой стоимостью.
6. Клиент подтверждает бронирование.
7. Система создаёт бронь через API и показывает подтверждение.

### Альтернативные потоки

- **A1. Недостаточно прокатной экипировки.** Система предлагает записаться только при использовании собственной экипировки.

### Исключения

- **E1. Нет свободных мест.** API возвращает отказ; система информирует клиента.

### Постусловия

- При успехе: создана бронь; запись видна в «Мои бронирования».
- При ошибке: бронь не создана.

## UC-005: Отмена бронирования клиентом

| Поле | Значение |
| :-- | :-- |
| **ID** | UC-005 |
| **Приоритет** | P1 |
| **Источник** | [DD], [CQ:16–18], [CQ:33] |
| **Связанные ФТ** | FT-018–FT-020 |
| **Актёр** | Клиент |
| **Предусловия** | У клиента есть активное бронирование; заезд ещё не начался |
| **Триггер** | Клиент решает отменить запись |

### Основной поток

1. Клиент открывает «Мои бронирования» и выбирает запись.
2. Клиент нажимает «Отменить».
3. Система определяет тип отмены по времени до старта.
4. **Ранняя отмена (≥ 2 ч до старта):** бронь отменяется, место освобождается.
5. Система показывает подтверждение отмены.

### Альтернативные потоки

- **A1. Поздняя отмена (< 2 ч до старта).** Система показывает предупреждение; после подтверждения фиксирует позднюю отмену без штрафа; место не освобождается.

### Постусловия

- Статус брони обновлён; при ранней отмене место доступно другим клиентам.

## UC-008: Обработка отмены заезда центром

| Поле | Значение |
| :-- | :-- |
| **ID** | UC-008 |
| **Приоритет** | P1 |
| **Источник** | [CB], [DD], [CQ:27] |
| **Связанные ФТ** | FT-021, FT-022, FT-030 |
| **Актёры** | Клиент, Существующий сервер |
| **Предусловия** | У клиента есть бронирование на заезд; центр отменяет заезд через существующую инфраструктуру |
| **Триггер** | Заезд отменён центром (в т.ч. из-за погоды) |

### Основной поток

1. Сервер изменяет статус заезда и бронирования.
2. Приложение получает обновлённый статус «Отменён центром» с указанием причины.
3. Система отправляет клиенту push-уведомление.
4. Клиент видит обновлённый статус в «Мои бронирования».
5. Повторная запись на этот заезд недоступна.

### Постусловия

- Клиент проинформирован; бронь сохранена в истории со статусом «Отменён центром».
