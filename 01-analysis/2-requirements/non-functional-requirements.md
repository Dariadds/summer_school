# Нефункциональные требования

> Этап 2. Как система должна работать: производительность, надёжность, безопасность,
> доступность, удобство. Частая ошибка ИИ — путать ФТ и НФТ.

> **Скоуп — клиентское мобильное приложение и API для него.** НФТ относятся к клиентскому
> приложению; функции инструктора/владельца/админки обеспечиваются существующей инфраструктурой.

| ID | Категория | Требование | Приоритет | Источник |
| :-- | :-- | :-- | :-- | :-- |
| NFR-1 | Удобство использования | Интерфейс клиента — mobile-first, рассчитан на использование с телефона на берегу (на солнце, мокрыми руками): крупные элементы, высокий контраст, минимум полей и шагов. | Высокий | бриф; описание домена |
| NFR-2 | Удобство использования | Запись на слот выполняется за минимальное число шагов (ориентир ≤3 экранов до подтверждения (предположение)), чтобы клиент мог записаться быстро и без обучения. | Высокий | бриф; ответы заказчика (Q20) |
| NFR-3 | Удобство использования | Лёгкая регистрация: только имя и телефон, без сложных паролей. Порог входа для клиента минимален. | Высокий | ответы заказчика (Q6) |
| NFR-4 | Платформа / Совместимость | Система работает в распространённых мобильных браузерах (последние версии Chrome, Safari) на типовых смартфонах без установки отдельного приложения (предположение). | Высокий | бриф; ответы заказчика (Q19) |
| NFR-6 | Производительность | Список доступных слотов и подтверждение записи откликаются быстро — ориентир: время отклика < 2–3 с при нормальной сети (предположение). | Высокий | бриф; описание домена |
| NFR-7 | Производительность | Система сохраняет приемлемую отзывчивость при пиковой нагрузке в выходные, когда одновременно записывается наибольшее число клиентов. | Высокий | бриф; ответы заказчика (Q20) |
| NFR-8 | Надёжность / Целостность данных | При одновременной записи нескольких клиентов на один слот исключаются двойные брони и овербукинг: количество занятых мест и прокатных досок никогда не превышает лимитов слота. Это критическое требование (главная боль заказчика). | Высокий (Must) | бриф; описание домена; ответы заказчика (Q1, Q2) |
| NFR-9 | Надёжность / Целостность данных | Состояние записей и свободных мест остаётся согласованным при сбоях и параллельных операциях (отмена освобождает место корректно, без потери или задвоения данных). | Высокий | бриф; описание домена |
| NFR-10 | Доступность (Availability) | Сервис доступен в часы работы клуба в течение сезона; целевой уровень доступности — порядка 99% в рабочее время (предположение). | Средний | описание домена |
| NFR-11 | Безопасность / Приватность | Персональные данные клиентов (имя, телефон) защищены от несанкционированного доступа; хранятся и передаются безопасно. | Высокий | описание домена (роли); ответы заказчика (Q6) |
| NFR-12 | Безопасность / Разграничение доступа | Клиент видит и управляет только своими записями; доступ к чужим записям и к данным управления (расписание, инструкторы) через клиентский API запрещён. | Высокий | ответы заказчика (Q4, Q14, Q15); описание домена |
| NFR-13 | Уведомления | Напоминания о записи доставляются своевременно (заблаговременно до старта прогулки), чтобы снизить неявки; доставку обеспечивает существующая инфраструктура. | Средний | бриф; ответы заказчика (Q16) |
| NFR-15 | Ограничения проекта | Решение должно быть запущено к началу сезона — жёсткий срок ~2 месяца. | Высокий (Must) | бриф; ответы заказчика (Q20) |
| NFR-16 | Ограничения проекта | Бюджет ограничен: применяется MVP-подход, приоритет простоты реализации; «навороты» (онлайн-оплата, авто-погода, лояльность) откладываются на последующие версии. | Высокий | бриф; описание домена (границы MVP) |
